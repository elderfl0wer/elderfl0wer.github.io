C style notes

These are the conventions I try to follow across my own C projects. They're not a formal standard — just the habits that keep a codebase readable to me a year later, and to anyone else who has to touch it.

Formatting

  • Tabs for indentation, treated as 8 columns wide.
  • Opening brace on the same line for control statements, own line for function definitions — the classic K&R split most long-lived C codebases converge on.
  • One statement per line; no hidden control flow behind macros where a function would do.
  • Lines wrapped well under 80 columns where practical.

Naming

  • lower_snake_case for functions and variables.
  • UPPER_SNAKE_CASE reserved for preprocessor constants and macros.
  • Names long enough to be unambiguous, short enough to read on one line — no Hungarian notation.

Practice

  • Every allocation has a clear, singular owner and a matching free.
  • Return values are checked; errors are handled at the point they occur, not deferred and forgotten.
  • Headers declare an interface, not implementation details — keep structs opaque where callers don't need the internals.
  • Compiler warnings are treated as bugs: build with -Wall -Wextra and keep the output clean.

For a much more thorough treatment of C style questions, the guides at invisible-island.net are worth reading in full.
